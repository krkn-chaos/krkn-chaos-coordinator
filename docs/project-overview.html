krkn-chaos-coordinator

AI-driven multi-agent system that finds gaps in chaos test coverage for OpenShift — and closes them automatically.

1 The Big Picture

🏥
Think of this system like a triage nurse for bugs. Thousands of bugs come in from JIRA. The system reads each one, decides which ones represent failure modes we should be proactively injecting in test environments, and then either writes the chaos test or files a ticket for a human to write it.

🐛 Input: Bugs & Regressions

Pulls from JIRA (OCPBUGS project) and Sippy (test regression data). These are real production failure reports from OpenShift clusters.

🤖 Processing: AI Agents

6 domain agents, each covering a part of OpenShift (control plane, networking, storage, etc.), run a multi-stage pipeline to analyze each bug.

🎯 Output: Tests & Issues

High-confidence gaps get draft PRs to krkn repos. Medium-confidence get GitHub issues. The goal: turn every uncovered failure mode into a chaos test.

2 Why This Exists

The Problem

  • OpenShift is massive — thousands of components, thousands of bugs filed every release cycle.
  • krkn is the chaos testing tool for OpenShift (like Netflix's Chaos Monkey for Kubernetes). It injects failures — kills pods, partitions networks, fills disks — to verify the platform recovers.
  • Nobody has time to manually read every bug and ask: "Should we have a chaos test for this?"
  • Coverage gaps pile up silently. Bugs describe real failure modes that happened in production, but if nobody creates a chaos test for them, we can't proactively detect regressions next time.

The Solution

Automate the entire workflow: read bugs, decide if they're chaos-testable, check if we already cover them, and create PRs/issues to close the gaps. Humans review and approve at the end — the AI does the legwork.

Key Insight Not every bug needs a chaos test. A typo in a log message, a CVE patch, or a UI rendering glitch — those aren't failure modes krkn can simulate. The filter is the most important part of the system.

Stage Details

1 DISCOVER

Queries JIRA for bugs in the agent's components. Prioritizes Critical/Major/Blocker bugs. Enriches with fix commits and Z-stream data. Default: last 14 days, up to 2000 bugs per agent.

2 FILTER

The most critical stage. Asks two questions: (1) Is this a resilience failure mode? (2) Can krkn inject this failure? Uses a 3-tier system: keywords → semantic cache → LLM. See the Filter section for details.

3 MAP

For chaos-relevant bugs, searches ChromaDB for existing krkn scenarios that already test this failure mode. Classifies as FULL_MATCH (already covered), PARTIAL_MATCH (similar but not exact), or NO_MATCH (gap found).

4 ANALYZE

For unmatched bugs (gaps), scores confidence 0-100. Gathers OCP docs, krkn docs, and similar resolved bugs from Neo4j. Suggests which base scenario to extend and what modifications are needed.

5 ACT

Based on confidence score: HIGH (70-100) creates draft PRs across krkn + krkn-hub + website repos. MEDIUM (40-69) creates a GitHub issue. LOW (0-39) logs it and skips.

6 REMEMBER

Stores everything in Neo4j knowledge graph — bugs analyzed, gaps found, actions taken. This prevents re-processing the same bug and enables trend analysis over time.

5 The Filter System (Most Important Part)

The filter decides whether a bug describes a failure that krkn can simulate. It's a 3-tier system designed to be fast, accurate, and cost-effective.

Tier 1: Keyword Pre-filter
Instant. Checks for chaos keywords (crash, panic, OOM, timeout) and anti-keywords (CVE, flaky test, docs).
High confidence → RETURN Ambiguous → continue
▼
Tier 2: Semantic Cache
1-5ms. ChromaDB embedding search — "have we seen a similar bug summary before?" If distance < 0.15, return cached result.
Cache hit → RETURN Cache miss → continue
▼
Tier 3: LLM Reasoning
3-5s. Claude/Ollama reads the bug + docs context and reasons: "Is this a chaos-testable failure mode?" Low-confidence results escalate to a stronger model.
Decision → RETURN + cache

6 Knowledge Layer

📖 ChromaDB Vector DB

What: Semantic search over docs and scenarios. Stores 4,089 embedded chunks across 3 collections.

Collections:

  • scenario_docs — 65 chunks from krkn scenario YAMLs
  • krkn_docs — 750 chunks from krkn website/hub docs
  • ocp_docs — 3,274 chunks from OpenShift module docs

Key feature: Per-component search — queries each component separately, merges and deduplicates, avoids dilution for multi-component bugs.

🕸 Neo4j Graph DB

What: Persistent knowledge graph tracking everything the system has analyzed.

Node types:

  • Bug — JIRA bug with all metadata
  • Component — linked to bugs via [:HAS_BUG]
  • Gap — coverage gaps linked to bugs via [:HAS_GAP]
  • Run — agent execution with metrics
  • Action — GitHub PRs/issues linked to gaps

Purpose: Deduplication (don't re-analyze bugs), trend analysis, tracking which gaps have been resolved.

7 Confidence Scoring & Actions

After identifying a coverage gap, the ANALYZE phase scores it 0-100. The score determines what action the system takes.

ScoreLevelActionWhat Happens
70-100 HIGH Draft PR Creates PRs across krkn + krkn-hub + website repos with scenario YAML, wrapper script, and docs
40-69 MEDIUM GitHub Issue Creates a GitHub issue with detailed recommendation, base scenario suggestion, and modifications needed
0-39 LOW Skip Logs the gap but takes no action — not enough confidence the gap is real or actionable
Human Review Required The system never merges PRs or closes issues on its own. Everything goes into an approval queue that a human reviews. The AI does the research and drafting — humans approve.

10 Questions You Might Ask

What is krkn exactly?
krkn is an open-source chaos engineering tool specifically built for Kubernetes/OpenShift. Think of it like Netflix's Chaos Monkey but for the Red Hat ecosystem. It can inject failures like killing pods, draining nodes, partitioning networks, filling disks, skewing clocks, and more. It lives at github.com/krkn-chaos/krkn. There's also krkn-hub (wrapper scripts for common scenarios) and the krkn website (documentation).
What is OpenShift and how does it relate to Kubernetes?
OpenShift is Red Hat's enterprise Kubernetes distribution. It adds features on top of vanilla Kubernetes — operators, a web console, integrated CI/CD, enhanced security, etc. The bugs we analyze come from OCPBUGS, the JIRA project tracking OpenShift bugs. When we say "component" we mean an OpenShift component like "kube-apiserver" or "Machine Config Operator."
Why do we need AI/LLM for this? Can't we just use keywords?
Keywords handle ~60-70% of cases well. But many bugs are ambiguous. For example: "API server returns 504 under high load" — is that a chaos-testable resource exhaustion issue, or just a configuration problem? The LLM reads the full bug description, understands context, and makes nuanced judgments. We use a tiered approach: keywords first (free, instant), then LLM only when needed (costs ~$0.01-0.05 per bug).
What are the domain agents and why 6 of them?
OpenShift has ~96 component areas. We split them into 6 logical groups that match how the engineering teams are organized. Each agent specializes in its domain and queries only its relevant components from JIRA. They run in parallel for speed and their results are deduplicated by the orchestrator. The 6 agents are: Control Plane, Upgrade/Lifecycle, Node/Machine, Networking, Storage, and Operators/Platform.
What's ChromaDB and why do we use it?
ChromaDB is a vector database — it stores text as numerical embeddings and lets you search by semantic similarity ("find docs that are about network partitions" rather than "find docs containing the word 'partition'"). We use it for three things: (1) searching krkn scenario configs to see if a test already exists, (2) pulling relevant documentation for LLM context, and (3) caching filter results for similar bugs.
What's Neo4j used for here?
Neo4j is a graph database. We use it to track relationships: Bug → has Gap → resolved by Action (PR/Issue). This lets us answer questions like "which components have the most unresolved gaps?" and "have we already analyzed this bug?" It's the system's long-term memory — preventing duplicate work across runs and enabling trend analysis.
What does a typical run look like? How long does it take?
A single agent run (e.g., Control Plane) takes ~30-90 seconds depending on LLM usage. It discovers 50-200 bugs from JIRA, filters most of them out (typically 70-80% are not chaos-relevant), maps the remaining against existing scenarios, and analyzes any gaps found. Output is a structured approval queue showing what actions the system recommends.
What LLM models does this support?
It's pluggable — supports Claude (via API or CLI), OpenAI, Google Gemini, and Ollama (local models). The system auto-detects what's available from environment variables. You can route different pipeline phases to different models (e.g., cheap/fast Sonnet for filtering, expensive/thorough Opus for analysis). If no LLM is configured, it falls back to keyword-only mode.
Does the system merge PRs or create issues automatically?
No. The system creates draft PRs and issues, then puts them in an approval queue. A human (that's us) reviews and decides whether to merge or close. The AI does the research and drafting work; humans make the final call. This is by design — we don't want autonomous PR merging for chaos test infrastructure.
How do I add a new domain agent?
  1. Create a new file in src/agents/ (e.g., my_agent.py)
  2. Inherit from BaseDomainAgent — it gives you the full pipeline for free
  3. Add your component mapping in src/knowledge/component_map.py
  4. Register the agent in AGENT_CLASSES in src/main.py
  5. Write tests in tests/unit/
You mainly just define which OCPBUGS components your agent covers.
What's the cost per run?
After recent optimization: ~$0.67 per full run (all 6 agents) with LLM. Without LLM (keyword-only mode): essentially free. The main cost drivers are LLM calls during FILTER and ANALYZE phases. We minimize costs with the 3-tier filter (keywords → cache → LLM) so the LLM is only called for ambiguous cases.
What's a "scenario" in krkn context?
A scenario is a YAML config file that tells krkn what failure to inject. For example, a pod disruption scenario might say: "In namespace X, kill pods matching label Y, wait Z seconds, verify they recover." Scenario types include: pod_disruption, node, network_chaos, resource_stress (hog), PVC (disk fill), time_skew, and cloud_provider. Each scenario maps to a krkn plugin that does the actual injection.
What's a "frozen dataclass" and why does everything use them?
Python's @dataclass(frozen=True) makes instances immutable — you can't change their fields after creation. We use them for all domain models (Bug, FilterResult, GapAnalysis, etc.) because: (1) it prevents accidental mutations as data flows through the pipeline, (2) frozen objects are hashable and safe to use as dict keys or set members, (3) it makes the data flow explicit — you always create new objects rather than modifying existing ones.
What should I work on first?
Start by running the unit tests and understanding the data flow. Then try a single-agent run with --agent control_plane and read the output. Good first contributions: (1) improving keyword lists in the filter, (2) adding tests for edge cases, (3) improving the approval queue formatting, (4) adding a new domain agent for components you're interested in.
What testing framework do we use?
pytest with pytest-cov for coverage. Tests are in tests/unit/ and tests/integration/. We target 80%+ coverage. Run with PYTHONPATH=. pytest tests/unit/ -v. Integration tests require Neo4j running (docker compose up -d neo4j).
